# ws-005: AI-Native Agent Fleet Protocol (SFP)

**Status:** planned
**Created:** 2026-03-15

## Goal

Real-time monitoring and management of 100K+ deployed SigmaShake AI agents via a purpose-built protocol. Uses Cloudflare Durable Objects with WebSocket Hibernation — no external MQTT broker. Agents maintain persistent bidirectional connections, publish structured telemetry (tool calls, LLM costs, session traces), and receive capability-scoped commands.

## Architecture Decisions

| Question | Decision | Rationale |
|----------|----------|-----------|
| Transport | Durable Objects + WebSocket Hibernation | GA product, $0 for hibernated connections, full control over auth/sharding, bidirectional |
| Wire format | JSON default, CBOR opt-in | Debuggable, serde-compatible, ~200 bytes/heartbeat. CBOR negotiated at handshake for high-volume tenants |
| Sharding | `fleet-{tenant_id}-{hash(agent_id) % shard_count}` | Strict tenant isolation at connection layer, ~100 agents per DO, auto-scales |
| Fleet state | DO-local (authoritative) + KV aggregate (dashboards) + D1 (historical) | Real-time from DO, eventual consistency for dashboards, D1 for audit trail |
| SDK strategy | Rust core (`fleet-client`) + napi-rs (Node) + PyO3 (Python) wrappers | Single implementation, guaranteed protocol correctness, no drift |
| Why not MQTT | IoT-oriented, no agent identity/capabilities, no structured telemetry, CF can't run a broker | Purpose-built protocol understands WIMSE tokens, delegation chains, tool call traces |

## Connection Lifecycle

1. Agent opens WS to `/v1/fleet/ws` with `Authorization: Bearer <WIMSE JWT>`
2. Fleet Worker verifies JWT via `AgentIdentityVerifier`, computes shard, forwards to DO
3. Agent sends `ClientHello` → DO responds `ServerHello` with session_id + intervals
4. Agent sends `Heartbeat` every 30s, `MetricsReport` every 60s, `ToolCallTrace` per call
5. Server sends `Command` messages (capability-scoped authorization)
6. On disconnect: DO marks agent `Disconnected`. After 3 missed heartbeats (90s): `Offline`

## Message Types

**Client → Server:** ClientHello, Heartbeat, MetricsReport, ToolCallTrace, AgentAlert, CommandAck
**Server → Client:** ServerHello, Command, ConfigUpdate, CapabilityChange

**Command Types:** Restart, UpdateConfig, RevokeCredentials, InjectTool, RemoveTool, Pause, Resume, CollectDiagnostics, UpdateVersion

## Phase 5: Fleet Dashboard UI — Details

### New Backend Endpoints (T15j)

Added to fleet-server, OpenAPI, and api-gateway router:

1. **`GET /v1/fleet/agents/:agent_id/metrics`** — D1 `agent_metrics` filtered by agent_id, time-series for sparklines (1h/24h/7d). Response: `{ metrics: [{ timestamp, cpu_pct, memory_mb, llm_tokens_in, llm_tokens_out, llm_cost_usd }] }`
2. **`GET /v1/fleet/agents/:agent_id/commands`** — D1 `fleet_commands` filtered by agent_id, paginated. Response: `{ commands: [{ id, command_type, payload, issued_at, acked, ack_message }] }`
3. **`GET /v1/fleet/config`** — KV `fleet:{tenant_id}:config`. Response: `{ heartbeat_interval_secs, metrics_interval_secs, max_agents, alert_thresholds: { missed_heartbeats, error_rate_pct }, auto_scale_enabled }`
4. **`PUT /v1/fleet/config`** — Write to KV, pushes `ConfigUpdate` to all connected agents via broadcast.
5. **`GET /v1/fleet/events` (SSE)** — Real-time event stream. Events: `agent_connected`, `agent_disconnected`, `agent_offline`, `alert_fired`, `command_sent`, `command_acked`.

### Frontend File Structure

```
src/pages/fleet/
  mod.rs              — FleetPage (tab container: Overview | Agents | Config | Commands)
  types.rs            — FleetStatus, AgentRow, AgentDetail, FleetConfig, FleetEvent, CommandRow
  fetch.rs            — fetch_fleet_status(), fetch_agents(), fetch_agent_detail(), etc.
  overview.rs         — Health cards + live event feed + status summary
  agents.rs           — Sortable DataTable + status filter + search
  agent_detail.rs     — Slide-out drawer: info, sparklines, command history, send command
  config.rs           — Config form: intervals, thresholds, auto-scale toggle
  commands.rs         — Broadcast command panel: type picker, payload editor, confirmation
  events.rs           — SSE stream component (fetch+ReadableStream, same as logs.rs)
```

### Patterns to Reuse (from existing sigmashake.com)

- **Auth:** `fetch_json_authed()` with Bearer + X-Tenant-Id headers (`src/api.rs`)
- **Layout:** AuthLayout wrapper with nav tabs (`src/app.rs`)
- **Components:** DataTable, StatusBadge, Skeleton, EmptyState, ErrorDisplay (`src/components/`)
- **State:** `PageState<T>` enum (Loading/Ready/Error), `use_state` + `use_effect_with` (`src/hooks.rs`)
- **SSE streaming:** fetch+ReadableStream pattern from logs page (`src/pages/logs.rs`)
- **Forms:** FormState enum (Idle/Submitting/Success/Error) (`src/types.rs`)
- **CSS:** `.dash-card`, `.dash-grid`, `.status-badge` patterns → mirror as `.fleet-*`
